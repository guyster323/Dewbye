# ESS HVAC 화재 위험 예측 앱 - 종합 설계 (참고 자료)

**프로젝트명**: ESS Guardian - HVAC 기반 습도 이상 징후 조기 경보 시스템  
**타겟 사용자**: ESS 설치/운영 기업, 에너지 저장시스템 관리자  
**상태**: 개념 단계 (화재와의 인과관계 추가 검증 필요)  
**연관 앱**: Weather-HVAC Analytics 기반 확장 가능

## 1. 개념 & 배경

### 1.1 ESS 화재의 환경적 요인

**배터리 화재의 직접 원인**:
- 내부 단락, 과전압, 과전류
- 불량 배치 또는 결함 부품

**환경 조건의 악화가 이들을 가속화**:
- 고온: 배터리 화학반응 가속, 내부 압력 상승
- 고습: 절연체 저하, 누전/단락 가능성 증가
- 급격한 환경 변화: HVAC 대응 불가, 스트레스 발생

### 1.2 HVAC-습도 상승의 특이점

**일반적인 HVAC 제어**:
```
외기 온도 ↑ → 냉방 강화 → 실내 온도 ↓
외기 온도 ↓ → 냉방 약화 → 실내 온도 ↑
```

**문제점: 모드 전환 시점**
- 외기 온도가 설정점(21°C)을 넘나드는 순간
- 냉방 ↔ 난방 사이 압축기 켜졌다 꺼짐
- **제습 능력이 일시적으로 저하**
- → 실내 습도 급상승 (임계점 발생)

**수학적 설명**:
```
외기: 습도 70% @ 28°C → 절대습도 15.6 g/m³
HVAC 냉방 전환 → 실내 온도 21°C로 강하
절대습도는 유지되므로
상대습도 = 15.6 / (포화값 @ 21°C) 
         = 15.6 / 18.5 = 84% (!)

↓

이슬점: 18.5°C
배터리 표면 온도 < 18.5°C → 결로 발생
절연 특성 급격히 악화
```

### 1.3 ESS 화재와의 추정 연관성

```
높은 습도 + HVAC 급격한 모드 전환
    ↓
배터리 절연체 표면 결로 발생
    ↓
절연 저항 급감 (특히 코팅 부분)
    ↓
누전 전류 증가
    ↓
배터리 내부 고온 발생
    ↓
화학반응 가속 → 가스 발생 → 폭발
```

**⚠️ 현황**: 학술적 검증 진행 중 (인과관계 아직 확정되지 않음)

## 2. 데이터 기반 접근

### 2.1 학술 근거 (4개 논문)

| 논문 | 결론 | 활용 |
|------|------|------|
| 제습냉방시스템 성능 | 습도 20%↑ → COP 30% 저하 | 제습 능력 한계 모델 |
| 액체식 제습 제어 | 모드 전환 시점 = ±1.5℃/hour | HVAC 신호 임계값 |
| 밀폐형 온실 | 기밀도별 습도 응답 다름 | ESS 용기 밀폐도 고려 |
| 온습도 동시제어 | 환기량 + 제습 함께 조절 필요 | 복합 제어 전략 |

### 2.2 현장 데이터 수집 (필요)

**ESS 현장에서 필요한 데이터**:
1. **외기 기상**: 온도, 습도 (기상청 API)
2. **HVAC 운용**: 냉방/난방 모드, 온도 설정값, 제습기 ON/OFF
3. **실내 환경**: 온도, 습도 (센서 필요)
4. **배터리 상태**: 온도, 절연 저항 (BMS 데이터)
5. **사건 기록**: 누전 차단기 작동, 화재 사건, 점검 기록

**가설 검증 방법**:
```
과거 ESS 화재 사건 3~5건 수집
  ↓
해당 날짜/지역의 기상 데이터 분석
  ↓
우리 알고리즘으로 위험도 계산
  ↓
"그 당시 경고했을까?" 검증
  ↓
정확도 > 70% 달성 → 신뢰성 확보
```

## 4. 현재 상태 및 한계

### 4.1 검증되지 않은 가설

```
✓ 과학적 근거 있음:
  • HVAC 모드 전환으로 습도 급변화 (논문 근거)
  • 이슬점 도달로 결로 발생 (기초 물리학)
  • 고습도로 절연 저항 저하 (전기공학)

? 그러나 ESS 화재와의 직접 인과관계는:
  • 대규모 현장 데이터 부족
  • 배터리 화학/전기적 복잡성
  • 다른 요인들과의 상호작용 미파악
  • 정량적 위험도 모델 미정립
```

### 4.2 필요한 검증 단계

```
1단계: 데이터 수집 (3개월)
  └─ 과거 ESS 화재 사건 10건 이상 수집
  └─ 각 사건의 기상 데이터, HVAC 로그, BMS 데이터
  └─ 당시 현장 점검 기록, 사후 분석 자료

2단계: 상관관계 분석 (2개월)
  └─ "화재 발생 직전 우리 알고리즘이 위험 경고했을까?"
  └─ 정확도, 오탐율, 미탐율 계산
  └─ 임계값 최적화

3단계: 전문가 검수 (지속)
  └─ 배터리 전문가 (화학공학)
  └─ 전기 안전 전문가 (전기공학)
  └─ ESS 운영업체 (실무)
  └─ 에너지 공단 (정책)

4단계: 학술 발표 및 논문화
  └─ 학술대회 발표
  └─ 국제 저널 투고
  └─ 특허 출원 (필요시)

5단계: 산업 표준화
  └─ ESS 안전 가이드라인에 반영
  └─ 정부 정책 제안
```

## 5. Weather-HVAC Analytics와의 관계

### 5.1 포지셔닝

```
Weather-HVAC Analytics (기본 앱)
├─ 기상 + HVAC 패턴 분석
├─ 타겟: 건물 에너지 관리자, 건설사
├─ 용도: 일반적 건물 환경 최적화
└─ 위험: 중간 (누전, 곰팡이, 부식)

    ↓ 확장/특화

ESS Guardian (특화 버전, 개념 단계)
├─ 기상 + HVAC + BMS 데이터
├─ 타겟: ESS 운영사, 에너지기업
├─ 용도: ESS 화재 위험 조기 경보
└─ 위험: 극고 (화재/폭발)
```

### 5.2 개발 전략 제안

**Option A: 순차 개발**
```
1. Weather-HVAC Analytics 먼저 출시 (MVP)
   ├─ 시장 검증
   ├─ 사용자 피드백 수집
   ├─ 수익화 시작

2. 1년 후 ESS Guardian 특화 버전 개발
   ├─ Weather-HVAC Analytics 기반
   ├─ BMS 데이터 연동 추가
   ├─ ESS-특화 위험도 알고리즘
```

**Option B: 병행 개발**
```
1. Weather-HVAC Analytics 개발 (3개월)
2. 동시에 ESS 데이터 수집 & 검증 (진행 중)
3. 2년차에 ESS Guardian 론칭
```

**권장사항**: **Option A** (순차 개발)
- 기본 앱으로 시장 기반 확보
- ESS 버전은 충분한 검증 후 개발
- 위험성 대비 신뢰성 우선
